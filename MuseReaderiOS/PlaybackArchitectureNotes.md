# MuseReader iOS Playback Bridge

## Current Playback Path

The Play button now prepares playback through `LiveScoreRenderSession` and `NativePlaybackController`.
The controller starts `AVAudioEngine` after the first short audio chunk is available, then continues scheduling chunked audio from the live score session. It no longer requires a full-score PCM buffer before playback can start.
Audio chunks are generated from MuseScore `PlaybackModel` / `PlaybackEventsRenderer` MPE note events for the active full score or excerpt. The Play path does not export MIDI and does not ask FluidSynth to render a whole MIDI file before playback.

User-facing MIDI export remains in `ScoreDetailView` and is separate from playback.

## Bridge Boundary

The iOS bridge surface is intentionally native and does not expose desktop QML concepts:

- prepare playback for the active `LiveScoreRenderSession`
- play, pause, stop, seek
- report state and time-aware position
- stream time-windowed audio chunks from the active full score or excerpt
- invalidate playback artifacts when score edits or part switches occur
- expose playback measure regions for score following

## MuseScore Dependencies

Required for first iOS playback:

- active `Score` or excerpt selection in `score_render_core`
- `mu::engraving::PlaybackModel` and `PlaybackEventsRenderer` event generation
- MPE `PlaybackData`, note events, arrangement timing, pitch context, and setup data
- playback measure regions from the same repeat/timing map for duration and score following
- FluidSynth as the current event-driven sampler backend, fed directly by scheduled note on/off events
- `AVAudioEngine` output and scheduled chunk streaming on iOS
- revision tokens on Swift-side invalidation to prevent stale chunks from replacing newer playback state

Required for higher fidelity later:

- MuseScore mixer track/channel abstractions
- sound profile and sampler resource selection
- articulation, expression, dynamic, and instrument-change routing
- edit-range invalidation from MuseScore playback cache semantics

Desktop/QML pieces to skip on iOS:

- QML playback toolbar and mixer panels
- desktop audio device preference UI
- project audio export dialogs
- desktop-only module wiring around playback UI state

## Playback Stack Audit

MuseScore proper starts playback in `src/playback/internal/playbackcontroller.cpp`, which coordinates notation playback models with `IPlayback`. `src/framework/audio/main/internal/playback.cpp` owns sequence setup at the audio framework boundary, and `src/framework/audio/engine/internal/engineplayback.cpp` drives engine sequences/tracks. Notation events are generated by `src/engraving/playback/playbackmodel.cpp` and `src/engraving/playback/playbackeventsrenderer.cpp`.

The iOS bridge boundary intentionally skips desktop QML/UI controller objects. It keeps the active render session as the source of truth, asks the active score/excerpt for MPE playback data, maps each MuseScore instrument track to a sampler channel, and streams short PCM chunks to `AVAudioEngine`.

## Removed From Playback Path

- MIDI export before playback
- FluidSynth MIDI player rendering from exported MIDI
- full-score PCM pre-render as the default Play button route
- full-buffer scheduling assumptions
